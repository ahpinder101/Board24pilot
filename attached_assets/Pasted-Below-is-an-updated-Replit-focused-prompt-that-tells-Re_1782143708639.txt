## P1 scope to implement now

### P1-1 — Formal answerability status on every response

Add a required field to every user-facing answer response:

```ts
answerability_status: "answerable" | "partially_answerable" | "not_answerable"
```

Rules:

* `answerable`: the answer is fully supported by retrieved evidence.
* `partially_answerable`: some requested parts are supported, but at least one part is missing, weak, ambiguous, conflicting, or uncertain.
* `not_answerable`: retrieved evidence does not support a meaningful answer.
* Never return a confident-sounding answer when evidence is weak.
* Every API response that answers a user question must include this field.
* Preserve any existing confidence field or badge, but do not treat confidence as a replacement for answerability.

---

### P1-2 — Structured `missing_or_weak_evidence` array on every response

Add this required response field:

```ts
missing_or_weak_evidence: Array<{
  claim_or_question_part: string;
  issue: "missing" | "weak" | "ambiguous" | "conflicting";
  explanation: string;
  related_citation_ids?: string[];
}>
```

Rules:

* If `answerability_status` is `answerable`, this array may be empty.
* If `answerability_status` is `partially_answerable` or `not_answerable`, this array must contain at least one item.
* Do not bury evidence gaps only in prose.
* For partial answers, show exactly which part of the user’s question could not be fully supported.
* If a citation is weak, missing, or only loosely relevant, add a corresponding item here.

---

### P1-3 — `citation_quality` field per citation

Extend each citation object with:

```ts
citation_quality: "strong" | "partial" | "weak" | "missing"
```

Suggested citation object shape:

```ts
{
  citation_id: string;
  evidence_id?: string;
  manual?: string;
  page?: number | string;
  excerpt?: string;
  supports_claim?: string;
  citation_quality: "strong" | "partial" | "weak" | "missing";
}
```

Rules:

* `strong`: the citation directly supports the claim with specific matching evidence.
* `partial`: the citation is relevant but does not fully prove the claim.
* `weak`: the citation is only loosely related, keyword-based, generic, or indirect.
* `missing`: no valid citation supports the claim.
* No citation may silently default to `strong`.
* If uncertain, classify as `partial` or `weak`, not `strong`.
* Any answer claim relying on `weak` or `missing` citation support must be reflected in `missing_or_weak_evidence`.

---

### P1-4 — Revise-once loop with second validation pass

Update the answer pipeline to enforce this flow:

```txt
retrieve evidence
→ draft answer
→ validate draft against evidence
→ if unsupported or weak claims are found, revise once
→ validate revised answer again
→ return final response
```

Add this metadata to the final response:

```ts
validation_metadata: {
  validation_pass_count: number;
  revised_once: boolean;
  final_validation_status: "passed" | "passed_with_warnings" | "failed";
}
```

Rules:

* If the first validation finds unsupported claims, revise the answer once.
* After revising, always run validation again.
* Do not create an infinite retry loop.
* Maximum one revision.
* If issues remain after the second validation pass, do not hide them.
* Remaining issues must affect `answerability_status` and must be listed in `missing_or_weak_evidence`.
* If the answer cannot be safely supported, return a guided no-answer or partial answer instead of overclaiming.

---

Add a visible selector to the question UI:

```ts

  | "auto"
  | "electrical"
  | "hydraulic"
  | "pneumatic"
  | "mechanical"
  | "troubleshooting"
```

UI label:

```txt
Domain area
```

Options shown to the user:

```txt
Auto
Electrical
Hydraulic
Pneumatic
Mechanical
Troubleshooting
```

Rules:

* Default value should be `auto`.
* Include the selected value in the request payload sent to the backend.
* Backend must accept and preserve this value.
* If the user selects anything other than `auto`, use that value to guide validation/routing instead of relying only on auto-detection.
* If existing domain routing is simple, add a safe placeholder now and leave clear TODO comments for later specialist validators.
* The frontend should display the selected or resolved domain somewhere in the result metadata if such a metadata area already exists.
